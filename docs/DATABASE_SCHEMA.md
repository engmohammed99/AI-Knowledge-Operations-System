# Database Architecture

The system utilizes a hybrid database approach. **Pinecone** acts as the Vector Database (storing the dense number arrays for similarity search), while **PostgreSQL** acts as the Source of Truth, storing the raw text chunks, document metadata, and AI insights.

This split architecture allows us to leverage vector math for retrieval without exposing the LLM to unstructured, unmanageable raw data.

### Schema Definition (Drizzle ORM)

#### 1. `documents`
Tracks the high-level files and external sources ingested by the system.
- `id` (UUID, Primary Key)
- `filename` (Text)
- `sourceType` (Enum: 'file_upload' | 'external_api')
- `status` (Enum: 'pending' | 'processed' | 'failed')
- `contentHash` (Text, Unique) - Used for deduplication.
- `createdAt` (Timestamp)

#### 2. `document_chunks`
Stores the actual text snippets that are fed into the LLM context window.
- `id` (UUID, Primary Key)
- `documentId` (UUID, Foreign Key -> documents.id)
- `content` (Text)
- `vectorId` (Text) - Maps directly to the corresponding ID in the Pinecone Vector Index.
- `createdAt` (Timestamp)

#### 3. `insights`
Stores the proactive business intelligence generated by the nightly Cron Job.
- `id` (UUID, Primary Key)
- `category` (Enum: 'issue' | 'decision' | 'conflict')
- `title` (Text)
- `description` (Text)
- `source_document_ids` (JSONB) - An array of UUIDs linking the insight to its specific source documents.
- `createdAt` (Timestamp)
